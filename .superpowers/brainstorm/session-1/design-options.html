整体设计方案

基于你的选择：网页应用、手动粘贴原文、需要 A-D 全维度反馈、混合练习模式、测试阶段简化 Key 处理

A

单文件网页应用（推荐）

一个 HTML 文件 + 内嵌 JS/CSS。用户粘贴文本、填入 Azure Key、点击录音，浏览器直接调用 Azure Speech SDK，返回发音评估结果。UI 标红/标黄问题词句，支持“全文 → 重点句 → 重点词”的混合练习循环。

优点

  • 构建最简单，双击就能打开
  • 无后端、无依赖
  • 迭代最快

缺点

  • Azure Key 在浏览器中可见
  • 只适合自己测试用
B

网页 + 本地代理

前端负责录音和展示，本地起一个极小的 Node/Python 服务作为代理。浏览器把音频发给本地服务，由它持有 Azure Key 并调用 Azure Speech。Key 不暴露给网页。

优点

  • Key 不泄露
  • 更接近真实产品架构

缺点

  • 要同时跑前端和后端
  • 部署/调试多一步
C

Mock 数据先行

先用模拟的发音评估数据把完整 UI 和练习流程跑通，包括标红、分数、句子/单词循环。等流程验证无误后，再把模拟层替换为真实的 Azure Speech 调用。

优点

  • 不需要 Azure Key
  • 最快验证交互

缺点

  • 反馈不是真实的
  • 后续需要一次“接真实 API”的改造

我的推荐：A。既然当前是测试阶段、追求最简路径，单文件网页应用能最快让你用上真实反馈；Key 暴露问题只在公开部署时才需要担心，本地测试完全可接受。
